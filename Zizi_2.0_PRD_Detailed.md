# zizi 2.0 详细需求与逻辑拆解文档 (V2.0)
## 一、 全局规则与环境定义
网络与 Loading 规范：由于后端部署在 Render，具有长达 50 秒的“冷启动”休眠唤醒时间。所有涉及后端的接口（登录、OSS Token获取、大模型分析）被触发时，前端必须立刻弹出防穿透的全局/局部 Loading 遮罩。超过 5 秒未响应时，Loading 文案需变更为：“AI正在努力苏醒中，请耐心等待哦~”。
权限规范：首次使用相机或麦克风前，必须有友好的引导文案，被拒绝后需提供明确的“如何去系统设置开启权限”的图文提示。
数据冗余规范：所有的识字行为（字、图 URL、时间戳、熟练度初始化为0），必须在后端落库，供未来的“游戏模块”调用。
## 二、 账号与登录模块 (Auth)
1. 登录功能
正常流程：用户输入 11 位手机号和密码 -> 点击登录 -> 前端校验格式 -> 发送请求 -> 验证成功 -> 本地存储 Token -> 跳转至“拍图识字”首页。
异常流程：
格式错误：失焦时提示“请输入正确的手机号”。
密码错误：Toast 提示“账号或密码错误”。
网络超时：超过 60s 未响应，提示“网络开小差了，请重试”。
2. 注册与找回密码功能
正常流程：点击注册/忘记密码 -> 进入表单页 -> 输入手机号 -> 设置新密码 -> 再次确认密码 -> 提交 -> 成功后自动跳转回登录页并填充手机号。
异常流程：
两次密码不一致：提交按钮置灰或点击报错。
账号已存在（注册时）：提示“该手机号已注册，请直接登录”。
账号不存在（找回时）：提示“该手机号未注册”。
## 三、 拍图识字模块 (核心模块 1)
1. 拍照与直传逻辑
正常流程：用户在首页点击“大眼睛” -> 唤起原生相机 -> 拍照或相册选图 -> 确认图片 -> 前端请求后端拿 OSS STS Token -> 前端直传图片至阿里云 OSS -> 获取到公开的 Image_URL -> 进入下一步逻辑。
异常流程：
权限拒绝：弹出 Modal 提示“Zizi 需要大眼睛（相机）才能看世界哦，请在设置中允许拍照权限”。
Token 获取失败 / OSS 上传失败：重试 1 次，若仍失败，阻断流程，提示“哎呀，图片没有传上去，请检查网络”。
2. AI 识字与故事匹配（⭐ 核心底层逻辑）
【关键前置数据】：当前用户的字库等级（如：Level 1, Level 2...），此数据作为入参发送给后端。
【正常流程拆解】：
第一步：画面特征提取 (qwen-vl-max)。后端将 Image_URL 发给大模型，提取画面中的主要名词、动词或形容词集合（例如：一只橘猫躺在窗台旁边的沙发上睡觉，提取词汇：[“窗户”,"猫", "睡觉", "沙发", "橘色", "可爱"]）。
第二步：字库降维匹配 (后端逻辑)。后端拿到上述词汇，与用户当前选择的【适龄字库】进行交集匹配。
匹配规则：优先匹配名词单字（如“猫”）。如果在该级别字库中没有“猫”，则退而求其次匹配“睡”，或者大模型提炼的基础字（如“小”、“大”）。
第三步：儿童化语境生成 (LLM)。后端确认核心字（假设为“猫”）后，再次约束 Prompt 生成解析。
Prompt 示例：“你现在是一个温柔的幼儿园老师。请结合图片内容（图片里有一只在沙发上睡觉的橘猫），教3岁的小朋友认‘猫’这个字。要求：1. 长度在 50 字以内。2. 必须包含‘猫’字。3. 语气生动可爱，带有鼓励性质。”
输出预期：“哇！是一只胖乎乎的橘猫，它正在沙发上呼呼大睡呢。你也来学小猫‘喵喵’叫一下吧！”
第四步：语音合成预加载 (qwen3-tts-flash)。后端将上述文字和“猫”字分别调用 TTS 生成两段 MP3 链接，打包返回给前端。前端进入结果页，开始隐藏加载音频。
异常流程：
图片违规/无法识别：模型返回安全拦截。前端兜底展示：“大眼睛没看清楚呢，请换个角度再拍一张吧！”
匹配不到字库字：当图片过于抽象，且字库极度限制时，系统兜底选择一个常用赞美字（如“好”、“棒”），大模型生成：“哇，你拍的图片太棒了！我们今天来学习‘棒’字吧！”
TTS 生成失败：结果页不展示播放按钮，仅展示文字。
3. 结果页交互逻辑
正常流程：
顶部展示正方形裁剪的照片。
下方展示核心单字大卡片，以及大模型生成的解析文本。
用户点击单字卡片，播放单字的 TTS 发音；点击文本区域或独立播放按钮，播放整段解析的 TTS 发音。
点击“继续玩”，清空当前状态，重新唤起相机。
## 三-A、相机取景框增强需求（优先级：低）

> ⚠️ **开发前置条件**：开发此模块前，必须与产品负责人详细确认以下所有需求细节和技术方案，不可自行开始开发。

### 背景与现状

当前拍图识字和绘本拍摄页面使用 `<input type="file" capture="environment">` 调用系统原生相机，取景框为纯装饰性 UI（虚线边框 + 图标提示），无实时画面预览，拍照后无裁剪处理。

### 需求 1：实时取景预览

**描述**：用户进入拍图识字或绘本拍摄页面时，取景框内应显示摄像头实时画面，替代当前的装饰性占位 UI。

**详细要求**：
- 使用 `getUserMedia({ video: { facingMode: 'environment' } })` 获取后置摄像头视频流
- 将视频流渲染到 `<video>` 标签中，填满取景框区域（`object-fit: cover`）
- 进入页面时自动启动预览，离开页面时释放摄像头资源（`stream.getTracks().forEach(t => t.stop())`）
- 拍图识字取景框比例：正方形（1:1）
- 绘本拍摄取景框比例：竖版（3:4）

**技术方案要点**：
- 放弃现有 `<input type="file" capture>` 方式，改为 `getUserMedia` + `<video>` + Canvas 截帧
- 需兼容 iOS Safari 和 Android Chrome（均需 HTTPS 环境）
- PWA 模式下需确保证书有效

### 需求 2：拍照后按取景框裁剪

**描述**：用户通过取景框拍照后，程序自动根据取景框的显示区域对拍摄画面进行裁剪，实现"所见即所得"。

**详细要求**：
- 拍照方式：Canvas `drawImage(video, ...)` 截取当前帧
- 裁剪逻辑：根据取景框在视频画面中的实际像素区域，使用 Canvas 9 参数版本的 `drawImage` 进行精确裁剪
- 输出格式：裁剪后的图片作为 `Blob` / `File` 对象，传递给后续处理流程（上传、AI 分析等）
- 裁剪后的图片应与取景框中用户看到的内容完全一致

**技术方案要点**：
- 视频流原始分辨率与取景框显示尺寸之间存在比例差异，需要做坐标映射计算
- 不同设备摄像头分辨率不同，裁剪参数需动态计算

### 需求 3：横竖屏自适应

**描述**：程序能识别手机当前的横屏/竖屏状态，取景框样式和裁剪逻辑随方向变化自动调整。

**详细要求**：
- 通过 `window.matchMedia('(orientation: portrait)')` 或 `screen.orientation` API 检测当前方向
- 竖屏时：拍图识字取景框 1:1，绘本拍摄取景框 3:4
- 横屏时：拍图识字取景框调整为 4:3 或 1:1（待确认），绘本拍摄取景框调整为 4:3（待确认）
- 方向切换时动画过渡平滑，不闪烁
- Canvas 截帧时的裁剪区域根据当前方向动态计算

**技术方案要点**：
- CSS 使用 `@media (orientation: portrait/landscape)` 做响应式布局
- JS 监听 `orientationchange` 或 `resize` 事件更新裁剪参数
- 与需求 2 共享坐标计算逻辑

### 待确认事项（开发前必须讨论）

1. **HTTPS 部署**：`getUserMedia` 在非 localhost 环境下必须 HTTPS，当前开发环境是否已配置？
2. **取景框横屏比例**：横屏时拍图识字和绘本拍摄的取景框比例具体是多少？需要 UI 设计稿。
3. **拍照按钮位置**：改为实时预览后，拍照按钮的交互方式是否调整？是在取景框下方还是叠加在画面上？
4. **相册选图**：当前 `<input capture>` 同时支持拍照和从相册选择，改为 `getUserMedia` 后，是否仍需保留"从相册选择"的入口？如果需要，需额外增加一个相册入口按钮。
5. **兼容性降级方案**：如果设备不支持 `getUserMedia`（如老设备或隐私模式），是否需要回退到当前的 `<input capture>` 方案？
6. **性能考量**：持续开启摄像头会增加耗电和发热，是否需要在用户 idle 一段时间后自动关闭预览？

## 四、 绘本模块 (核心模块 2)
1. 绘本扫描与创建流
正常流程：
用户在“我的绘本”点击“添加”。
进入连拍模式：用户拍一页 -> 图片暂存本地数组 -> 屏幕显示“已拍 1 页，可继续拍摄” -> 用户继续拍 -> 用户点击“结束拍摄”。
批量上传：前端向后端请求 Token -> 批量将图片直传 OSS -> 拿到一组按顺序排列的 Image_URL_List。
进入“编辑状态”：后端起一个异步任务，调用 Qwen 构建故事。前端显示“AI正在施展魔法写故事...”。
异常流程：
单次拍摄超过 20 张：弹窗拦截“绘本太长啦，请分册拍摄哦”。
中途退出：弹窗二次确认“图片还未保存，确定要放弃吗？”
2. 绘本故事生成逻辑（⭐ 核心底层逻辑）
正常流程（后端 AI 编排流）：
整书连贯性抓取：将所有图片统一发送给大模型（或提供上下文），让其识别绘本的主角和主线剧情，并生成一个【绘本标题】。
分页剧本生成：遍历每一页图片，结合整书主线，生成该页对应的故事旁白（强制要求：受众为3-6岁，语言简短，富有童趣）。
批量语音合成：将每一页的剧本调用 TTS 生成 MP3 链接。
最后，将书名、页码、图片URL、剧本文本、音频URL 组合成 JSON 落库，并返回给前端展示。用户可修改标题。
异常流程：如果某页图片模糊导致大模型无法识别，AI 将根据前后页的剧情进行逻辑脑补“衔接语句”，保证故事不断连。
3. 绘本播放交互流
核心 UI：水平 Swiper 滑动容器，当前页 scale(1.1) 并居中，两侧页 scale(0.9) 并带有模糊/半透明遮罩。
滑动播放模式 (默认行为)：
正常流：用户停留在第 N 页 -> 启动 500ms (0.5秒) 的防抖定时器 -> 计时器结束 -> 播放该页音频。
中断流：如果在 0.5 秒内用户滑到了 N+1 页，则取消第 N 页的定时器，重新为 N+1 页倒计时。若音频正在播放时用户滑动，立刻打断当前音频，并在 0.5 秒后播放新页面音频。
自动连播模式：
正常流：用户点击“自动连播” -> 系统接管控制权 -> 播放第 N 页音频 -> 监听到音频播放完毕 (onended) -> 代码触发页面滑向 N+1 页 -> 延迟 0.5 秒 -> 播放 N+1 页音频。
异常/退出流：在连播期间，如果用户手指触摸并滑动了屏幕，或者点击了暂停按钮，系统立刻退出连播模式，恢复为手动滑动模式。
4. NFC 实体交互流
正常流程：家长在编辑页复制该绘本的专用 URL (https://xxx.com/book/123?autoplay=1) 并使用第三方工具写入实体 NFC 贴纸。手机（已解锁状态）碰触贴纸 -> 浏览器被唤起并打开该 URL -> 路由解析到带有 autoplay=1 参数 -> 自动跳过列表页，进入播放视图，并强制触发自动连播模式。
## 五、 设置与配置模块 (Settings)
1. UI 与交互流
正常流程：点击右上角齿轮 -> 屏幕右侧滑出 Drawer 遮罩层 -> 内部包含配置表单 -> 点击空白处或返回按钮收起。
2. 参数项逻辑
字库配置 (Level 1 - 4)：
单选逻辑。切换后存入 localStorage 和前端全局 Store。
联动逻辑：每次触发“拍图识字”或未来的“游戏识字”时，必须将该 Level 值放在 Request Payload 中传给后端，以约束 AI 的选字范围。
API 双轨制与音色配置：
输入框：允许填入阿里云大模型和 TTS 的 API Key。
单选框：选择 TTS 音色（如：知性女声、童声）。
联动逻辑：前端封装的 Axios 拦截器会在每次发请求前检查。如果填写了自定义 Key 和模型参数，则将这些数据序列化放入 HTTP Header (例如 X-Custom-LLM-Key)。后端如果检测到 Header 中有值，优先使用该 Key 实例化大模型；如果没有，则 fallback 读取后端 .env 里的系统默认 Key。
