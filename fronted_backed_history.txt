⏺ 本地运行步骤：

  1. 启动前端开发服务器
  npm run dev
  浏览器打开 http://localhost:5173

  ---
  2. 需要配套后端

  前端默认连接 http://localhost:3000/api（见 .env）。目前所有 API
  调用都依赖后端，包括：

  ┌──────────────────────────┬───────────────────┐
  │           接口           │       说明        │
  ├──────────────────────────┼───────────────────┤
  │ POST /auth/login         │ 登录              │
  ├──────────────────────────┼───────────────────┤
  │ GET /oss/sts-token       │ 获取 OSS 上传凭证 │
  ├──────────────────────────┼───────────────────┤
  │ POST /photo-word/analyze │ 图片识字          │
  ├──────────────────────────┼───────────────────┤
  │ GET /picture-book        │ 绘本列表          │
  ├──────────────────────────┼───────────────────┤
  │ POST /picture-book       │ 创建绘本          │
  └──────────────────────────┴───────────────────┘

  ---
  3. 没有后端时的两种方案

  方案 A：Mock 数据（推荐快速预览 UI）

  在 vite.config.ts 加一个 mock 插件，或者直接临时把 .env 改成：
  VITE_API_BASE_URL=https://your-deployed-backend.com/api

  方案 B：手机测试（需要 HTTPS + 摄像头）

  npm run dev -- --host
  然后手机访问 http://192.168.x.x:5173（本地网络）。
  注意：摄像头 API 在非 HTTPS 下只能 localhost 访问，手机测试需要 ngrok
  或正式部署。

  ---
  4. 在设置里填入 API Keys（绕过后端转发）

  应用内「设置抽屉」→「自定义 LLM Key / TTS Key」栏，可以填入 DashScope
  key，前端会通过 X-Custom-LLM-Key / X-Custom-TTS-Key
  请求头直接传给后端，后端直接透传即可。

     Zizi Backend + Admin SPA Plan

     Overview

     Build a NestJS backend serving the Zizi frontend API, plus a standalone Vue
      3 admin SPA. Both in a new monorepo directory
     /Users/martinlok/Projects/zizi_server.

     Repository Structure

     zizi_server/
     ├── backend/          # NestJS API
     │   ├── src/
     │   │   ├── auth/
     │   │   ├── oss/
     │   │   ├── photo-word/
     │   │   ├── picture-book/
     │   │   ├── admin/           # admin-only endpoints
     │   │   ├── users/
     │   │   ├── config/          # global config (API keys, limits)
     │   │   └── common/          # guards, decorators, interceptors

     Backend Tech Stack

     - NestJS + TypeScript
     - Prisma ORM (cleaner TS support than TypeORM)
     - PostgreSQL (Neon or Render Postgres)
     - JWT via @nestjs/jwt + passport-jwt
     - Alibaba Cloud OSS SDK for STS token generation
     - Axios for DashScope API calls (qwen-vl-max, qwen3-tts-flash)
     - class-validator + class-transformer for DTOs

     Database Schema (Prisma)

     model User {
       id          Int      @id @default(autoincrement())
       phone       String   @unique
       password    String   // bcrypt hashed
       role        Role     @default(USER)
       isBanned    Boolean  @default(false)
       createdAt   DateTime @default(now())
       books       Book[]
       photoWords  PhotoWord[]
     }

     enum Role { USER ADMIN }

     model Book {
       id        Int        @id @default(autoincrement())
       title     String
       coverUrl  String
       createdAt DateTime   @default(now())
       userId    Int
       user      User       @relation(fields: [userId], references: [id])
       pages     BookPage[]
     }

     model BookPage {
       id       Int    @id @default(autoincrement())
       pageNum  Int
       imageUrl String
       audioUrl String @default("")
       story    String @default("")
       bookId   Int
       book     Book   @relation(fields: [bookId], references: [id], onDelete:
     Cascade)
     }

     model PhotoWord {
       id          Int      @id @default(autoincrement())
       imageUrl    String
       word        String
       explanation String
       vocabLevel  Int
       createdAt   DateTime @default(now())
       userId      Int
       user        User     @relation(fields: [userId], references: [id])
     }

     model AppConfig {
       key   String @id  // e.g. "llm_api_key", "tts_api_key",
     "oss_access_key_id"
       value String
     }

     Backend API Endpoints

     Auth (public)

     - POST /auth/login → { phone, password } → { token }
     - POST /auth/register → { phone, password } → 201
     - POST /auth/reset-password → { phone, password } → 200

     OSS (JWT required)

     - GET /oss/sts-token → generate STS credentials from AppConfig OSS keys → {
      accessKeyId, accessKeySecret, securityToken, bucket, region, prefix }

     Photo-Word (JWT required)

     - POST /photo-word/analyze → { imageUrl, vocabLevel } → { word, 
     explanation, wordAudioUrl, textAudioUrl, imageUrl }
       - Reads X-Custom-LLM-Key header (fallback: AppConfig llm_api_key)
       - Reads X-Custom-TTS-Key / X-TTS-Voice headers similarly
       - Calls qwen-vl-max for word+explanation, qwen3-tts-flash for audio
       - Saves record to PhotoWord table

     Picture-Book (JWT required)

     - GET /picture-book → list user's books (no pages)
     - GET /picture-book/:id → single book with pages
     - POST /picture-book → { imageUrls[] } → creates Book + BookPages, calls AI
      for stories + TTS

     Admin (JWT required + ADMIN role)

     - GET /admin/users → paginated user list with stats
     - PATCH /admin/users/:id/ban → toggle ban
     - GET /admin/books → all books across users
     - DELETE /admin/books/:id
     - GET /admin/photo-words → all photo-word records
     - GET /admin/stats → counts, daily active users, usage totals
     - GET /admin/config → get all AppConfig entries
     - PUT /admin/config → { key, value }[] → bulk update

     API Key Resolution Logic

     For every AI call, the backend resolves the key in priority order:
     1. X-Custom-LLM-Key / X-Custom-TTS-Key request header (user's own key)
     2. AppConfig.llm_api_key / AppConfig.tts_api_key (global default from DB)

     Admin SPA Tech Stack

     - Vue 3 + Vite + TypeScript + Pinia + Vue Router
     - TailwindCSS with same design tokens as Zizi frontend
     - Same ui_design_spec.md design language
     - Axios with admin JWT token
     - Simple table/card components (no heavy admin framework)
